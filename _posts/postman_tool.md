Postman
========


Postman은 단순한 API 호출 도구가 아니라, **클라이언트 입장에서 서버를 직접 두드려보며 동작을 검증하는 가장 현실적인 수단**이다. UI 뒤에 가려진 실제 API 요청·응답을 분리해서 확인할 수 있고, 파라미터 하나만 바꿔도 서버가 어떻게 반응하는지 즉시 확인할 수 있다.

SQA 관점에서 Postman의 핵심 가치는 “UI 없이도 서버의 의도를 검증할 수 있다”는 점이다. UI 테스트는 화면 기준의 결과만 확인하지만, Postman을 사용하면 **요청 구조, 인증 방식, 상태 코드, 응답 스키마까지 개발자 시선에서 직접 검증**할 수 있다.

특히 클라우드 콘솔, 관리형 서비스, 내부 어드민 UI 테스트에서는

- UI는 정상인데 API는 비정상인 경우
- 특정 파라미터 조합에서만 오류가 발생하는 경우
- UI로는 만들기 힘든 요청 조건
    
    을 검증해야 하는 상황이 자주 발생한다. 이때 Postman은 **UI 테스트의 사각지대를 메워주는 기본 도구**가 된다.
    

---

## 1. API 단독 호출로 UI 의존성 제거하기

### 개요

Postman의 가장 기본적인 사용 방식은 UI를 거치지 않고 API를 직접 호출하는 것이다. 이를 통해 UI 로직과 서버 로직을 분리해서 문제를 바라볼 수 있다.

### 주요 활용 시나리오

- UI에서는 막혀 있는 요청을 API로 직접 호출해 서버 동작 확인
- UI 오류인지, API 오류인지 원인 빠르게 분리
- 동일 API를 여러 조건으로 반복 호출하여 서버 안정성 확인

### SQA 관점 포인트

- UI 버그로 보이던 이슈가 실제로는 API 문제인지 즉시 판단 가능
- 개발자와 대화 시 “UI에서 안 됩니다”가 아니라 “이 요청에서 이 응답이 옵니다”로 설명 가능
- Selenium 실패 케이스를 API 단위로 빠르게 재현 가능

---

## 2. 환경 변수와 토큰을 활용한 인증 흐름 검증

### 개요

Postman의 Environment와 변수 기능을 활용하면 로그인 → 토큰 발급 → API 호출 흐름을 실제 서비스 구조와 거의 동일하게 구성할 수 있다.

### 주요 활용 시나리오

- 로그인 API 호출 후 access token 저장
- 토큰 만료, 잘못된 토큰, 권한 없는 토큰으로 API 호출
- 계정/권한별 응답 차이 검증

## 3. Request / Response 구조 기준으로 API 스펙 검증

### 개요

Postman은 단순히 응답이 오는지만 보는 도구가 아니라, **요청과 응답이 스펙에 맞는지 확인하는 도구**다. 실제로는 이 관점이 SQA에서 가장 중요하다.

### 주요 활용 시나리오

- 필수 파라미터 누락 시 서버 응답 확인
- 잘못된 타입, 범위 값 전달 시 에러 처리 검증
- 상태 코드와 에러 메시지가 스펙과 일치하는지 확인

### SQA 관점 포인트

- API 문서와 실제 구현 간 괴리 조기 발견 가능
- UI에서는 가려지는 서버의 방어 로직 확인 가능
- “정상 동작”의 기준을 화면이 아닌 API 스펙으로 정의 가능
